🧭 PR Orientation — My Personal Take 🤖 AI-generated

This is not official documentation. The official contributing guide is at CONTRIBUTING.md. Read that first.

What follows is my personal read on how the PR process actually works in practice — not rules, just patterns I've observed as a contributor who cares about the project's health. I'm rainsupreme, a full-time engineer who works on Valkey. These are my opinions, not the project's.

✅ What makes a PR land quickly 🤖 AI-generated

Keep it small and focused

One thing per PR. Reviewers are busy; a 50-line fix gets reviewed in days, a 2000-line refactor sits for months. If you have a big change, ask yourself: can I split this into a preparatory refactor + the actual change? Usually yes, and both halves move faster as separate PRs.

CI green before requesting review

Broken CI is an immediate context-switch cost for reviewers. They open the PR, see red, and move on. Fix it first. If CI is flaky (and it is — see the analysis page), explicitly note "this failure is pre-existing flakiness in X test, unrelated to my change" with evidence.

Write a clear description

Answer three questions: what changes, why this change is correct, and how to verify it. Don't make the reviewer reverse-engineer your intent. A good PR description is a gift to the reviewer.

Link to prior context

For anything non-trivial: is there an open issue? A prior discussion thread? "Continues discussion from #NNNN" or "Fixes the bug described in #NNNN" tells reviewers there's context they can read. It also shows you did your homework.

Add the release-notes label yourself

The bot will block your PR if it doesn't have either release-notes or no-release-notes. Just add it yourself when you open the PR. Bugfixes and user-facing changes get release-notes. Refactors, test fixes, and internal plumbing changes get no-release-notes. When in doubt, add release-notes.

⚠️ What makes a PR die quietly 🤖 AI-generated

Going quiet after feedback

If a reviewer comments and you don't respond within a few weeks, the PR is effectively dead. The reviewer has moved on. Even a "thanks, working on it" keeps the thread alive. If life got busy, a brief "I'll get back to this" is better than silence.

Scope creep mid-review

If a reviewer asks for a small fix and you come back with three unrelated cleanup commits, they have to re-review from scratch. Make the requested changes and nothing else. File a follow-up PR for the other things you noticed.

Duplicating an existing open PR

Search before submitting. gh search prs --repo valkey-io/valkey --state open "your keywords" takes 30 seconds. The zrangebyscore bug had 3 open PRs simultaneously — none of them got extra attention for it, they just fragmented reviewer energy.

Missing the release-notes label

See above. The bot will block it and it'll sit in a gray state until someone notices.

🟡 Understanding major-decision-pending 🤖 AI-generated

The major-decision-pending label means the community hasn't reached consensus on whether to take the feature at all. It's not a blocker on your end — your code is fine. It means the project is still debating the direction.

These can take weeks to months to resolve. Currently 38 open PRs carry this label, some for over a year. Patience is the primary tool. Pinging maintainers won't speed up the decision — the decision process has its own timeline.

What can help: if there's a public discussion thread, contribute to it thoughtfully. Real-world use cases and data are more persuasive than enthusiasm. If there's no thread, starting one in GitHub Discussions can sometimes unblock things.

🔍 Finding the right reviewer 🤖 AI-generated

There's no formal reviewer assignment for most PRs. Maintainers pick up what they have context on. If your PR sits 3+ weeks with no activity, a polite ping is appropriate.

To find who to ping: git log --follow -n 20 -- path/to/file.c shows who recently touched the files you changed. Cross-reference that with who's been active recently (reviewing, commenting on other PRs).

Don't tag someone who committed to that file 2 years ago and hasn't been seen since. Active reviewers are worth more than domain experts who've moved on.

🌱 First contributions 🤖 AI-generated

The project is welcoming but bandwidth-constrained. Your first PR might take longer than you expect. That's not a signal about the quality of your work.

A good first PR to build trust: fix a real bug (not a typo), keep it under 100 lines, include a test. Show you understand the codebase conventions before proposing bigger changes.

The quick-win or good first issue labels (if they exist) are worth looking at for ideas. Small fixes with CI-green status are the fastest path to getting your name known in the project.
