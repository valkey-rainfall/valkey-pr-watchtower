⚠️ Anti-Patterns to Avoid

1. Fixed `after N` / SIGSTOP-SIGCONT timing instead of condition waits

Why it flakes: Races against async completion (RDB child, replica drain). Passes on fast runners, fails under load/valgrind.

Seen in: integration/replication.tcl (diskless rdb pipe), io-threads.tcl, dual-channel-replication.tcl

Fix: Replace with wait_for_condition / wait_for_ofs_sync / key-load-delay.

2. One hard-coded timeout regardless of build mode

Why it flakes: valgrind is 10-50x slower and ASan ~2-3x; a single fixed budget guarantees timeouts on instrumented jobs.

Seen in: memefficiency.tcl perform_defrag_test (fixed 30s = 150x200ms), other.tcl io-threads runtime modifiable

Fix: Scale timeout by $::valgrind, or gate with {needs:!valgrind}/{needs:speed}.

3. Exact numeric assertions on values that legitimately vary

Why it flakes: Allocator/platform/runner-noise makes exact equality or tight ranges flip nondeterministically.

Seen in: maxmemory.tcl slave buffer (cmd_count/2 delta), memefficiency.tcl efficiency thresholds, cluster gossip count range, other.tcl used_active_time<1s

Fix: assert_range with a tolerance floor; widen margins; assert monotonic direction not exact value.

4. Implicit cross-test state coupling without convergence guards

Why it flakes: A test assumes slot/cluster state propagated by a prior test is already visible; gossip lag breaks it.

Seen in: cluster/slot-migration.tcl, cluster/many-slot-migration.tcl

Fix: Add explicit wait_for_slot_state / per-node cluster_state==ok guard before asserting.

5. `wait_for_log_messages` with single-shot (1 1) budget

Why it flakes: Checks once; misses if the log line has not flushed yet.

Seen in: dual-channel-replication.tcl, replication.tcl

Fix: Give a real retry budget (e.g. 50 100ms) matching other waits.

6. Multi-node clusters with default spawn/node-timeout under sanitizer/TLS

Why it flakes: 5-10 node clusters + 1000ms node-timeout are too tight for instrumented CI; spawn timeouts.

Seen in: cluster/failover.tcl, cluster/failover2.tcl, faster-failover.tcl

Fix: Raise node-timeout and harness spawn timeout for heavy (ASan/TLS) configs.

7. Unpinned rolling-release CI targets + no continue-on-error for known-unstable platforms

Why it flakes: Fedora Rawhide is a rolling release; it breaks unpredictably and is the only currently-ACTIVE toolchain failure.

Seen in: test-fedorarawhide-tls-module, test-fedorarawhide-tls-module-no-tls

Fix: Pin the image digest, or mark continue-on-error:true so it informs without blocking.

🔬 Methodology & Limitations

Pipeline: daily-dashboard JSON → consolidated per-test dataset → 5 parallel subagents deep-analyzed each test (reading real .tcl bodies) → cross-cutting synthesis. Failure modes and gtest verdicts were assigned from the actual test bodies, recent error text, and which CI job each test fails on (valgrind/sanitizer/TLS reveal the condition).

Limitations: rich error text existed for only the 12 tests active in the recent 14-day window; older tests were classified from name + .tcl body + failing jobs. Blame/compare data was unavailable. Hit counts are cumulative over 466 runs — a high absolute count at a low rate reflects long observation, not a hot fire.
